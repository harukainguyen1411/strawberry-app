Current State vs Target State
Current State
  • Double-deploy conflict
    myapps-prod-deploy.yml races with release.yml on every merge. Both fire simultaneously, unpredictable result.
  • No env validation
    Missing VITE_FIREBASE_* vars produce silent broken builds that deploy successfully but fail at runtime. No CI check catches this.
  • Direct to production
    Every merge goes straight to live channel. No staging, no smoke tests, no gate. A broken build goes live immediately.
  • No smoke tests
    Zero automated validation against the deployed URL. Build passing in CI says nothing about the live app functioning.
  • No version.json
    Impossible to know what's deployed without reading CI logs. No external observability into the running artifact.
  • Turbo cache ignores env vars
    Cache hits can return a build that used different env vars than the current run. Stale secrets can slip through undetected.
Target State
  • Single deploy owner
    release.yml is the sole production deploy pipeline. Old conflicting workflow deleted in PR 1. No more race conditions.
  • Fail-fast env validation
    Vite plugin checks all VITE_FIREBASE_* at build time. Missing vars = build error, not a silent runtime failure.
  • Staging buffer + manual gate
    Every release goes through staging first. Smoke tests run. Manual approval required before anything touches the live channel.
  • Smoke tests at two gates
    Playwright tests block promotion from staging and trigger auto-rollback after production deploy. Broken deploys self-correct.
  • Version fully observable
    /version.json on every app. Deploy tags in git. Sentry release tracking. Discord notifications. Full audit trail.
  • Turbo cache hashes env vars
    turbo.json updated so VITE_* vars are part of the cache key. Cache hits are always valid for the current env config.
